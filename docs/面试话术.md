## 技术亮点（面试必讲，选2-3个）

### 1. 混合推荐算法（核心亮点）
> 实现了 UCF（基于用户的协同过滤）+ ICF（基于物品的协同过滤）+ SVD 矩阵分解三种策略加权融合。
>
> **公式**：`final_score = 0.3 × UCF_score + 0.3 × ICF_score + 0.4 × SVD_score`
>
> - **UCF**：构建用户-物品评分矩阵 → 余弦相似度找相似用户 → 推荐相似用户喜欢的物品
> - **ICF**：物品特征向量（类别one-hot + 价格区间 + 销量 + 评分）→ 余弦相似度 → 推荐相似物品
> - **SVD**：SGD 分解到 k=5 的隐向量空间 → 预测用户对未交互物品的评分
>
> **复杂度**：O(U×I)，U=用户数，I=物品数
> **冷启动**：新用户回退热门推荐，新物品回退基于内容的推荐
> **为什么选这三种？** UCF 解决"和你相似的人喜欢什么"，ICF 解决"这东西和那东西像不像"，SVD 解决数据稀疏问题。三者互补。

### 2. 抢单并发控制（高频考点）
> **问题**：100个骑手同时抢一个订单，如何防止超卖？
>
> **方案**：使用云数据库的 `where({status: "pending"}) + update()` 原子操作。
> - 更新条件中带 `status: "pending"`，只有状态匹配的文档才会被更新
> - `updateResult.stats.updated === 0` 表示抢单失败（已被别人抢走）
> - 无需事务、无需锁，云数据库单文档操作天然原子性
>
> **验证**：用 Python 多线程模拟 100 个骑手同时抢单，验证无一超卖。
>
> **对比**：如果用传统 MySQL 需要 `SELECT ... FOR UPDATE` 行锁，云开发 NoSQL 的原子更新更简洁。

### 3. 配送顺路匹配算法
> **多因素动态评分**：
> ```
> raw_score = 0.45 × distance_score + 0.25 × time_urgency + 0.15 × route_quality + 0.15 × load_factor
> ```
> - **distance_score**：Haversine 公式计算绕路率 `detour_ratio = actual_distance / direct_distance`
> - **time_urgency**：等待时间越久紧迫度越高 `min(wait_minutes / 30, 1)`
> - **route_quality**：高峰期 0.6，非高峰期 0.95
> - **load_factor**：骑手当前订单数越多得分越低 `1 / (1 + current_orders)`
>
> **组合优化**：贪心选 Top-K 候选 → 2-opt 局部搜索优化 → 随机扰动跳出局部最优
> **冷启动**：DBSCAN 聚类（eps=0.5km）→ 聚类感知推荐

### 4. AI 助手集成
> 接入豆包 LLM API，实现智能问答。
> - 上下文感知：记住对话历史，支持多轮对话
> - 动作路由：用户说"帮我下单"自动跳转下单页
> - Prompt 模板管理：客服、推荐、润色不同场景不同 prompt
> - 流式响应：SSE 实现打字机效果

### 5. 工程化实践

## 面试追问 - 满分回答（扩展版）

### Q: 推荐算法为什么选 SVD 而不是 ALS？
> 当时选 SVD 是因为数据量预估不大（校园场景，千级用户），SGD 方式灵活易于实现和调试。ALS 更适合大数据量并行计算（Spark生态），对于云函数这种无服务架构，SVD + SGD 更轻量，k=5 的隐向量精度和计算开销平衡得很好。

### Q: 2-opt 优化具体怎么实现的？
> 初始解用贪心策略得到一条顺序路线。然后不断尝试交换路线中任意两条边的顺序（即 2-opt 交换），如果新的路线总距离更短就接受。反复迭代直到无法改进或达到最大迭代次数。相比穷举 O(n!)，2-opt 多项式时间内就能得到接近最优的解。

### Q: 数据库设计怎么考虑的？
> 主要集合包括：users、boxes、orders、deliveries、riders。关键索引有：`orders.status`（抢单查询）、`boxes.category + status`（分类浏览）、`deliveries.riderOpenid + status`（骑手订单列表）。后续数据量大了设计了分库分表方案（ShardingSphere），详见文档。

### Q: Redis 用在哪些地方？
> 1. **热门盲盒缓存**：首页热门榜单，5分钟过期
> 2. **用户 Session**：登录态管理
> 3. **推荐结果缓存**：用户的推荐列表，1小时过期
> 4. **抢单防重复**：用 SETNX 做分布式锁（兜底方案）
> 5. **LRU 淘汰策略**：防止缓存雪崩

### Q: 如果让你重做这个项目，你会改进什么？
> 1. 初期就引入 TypeScript，类型系统对云函数开发帮助很大
> 2. 推荐算法引入 Graph Embedding（如 Node2Vec），更好捕捉用户行为序列
> 3. 引入单元测试覆盖率门禁（>=80%）
> 4. 支付流程对接真实微信支付（目前是模拟）
> 5. 消息推送从轮询改成 WebSocket 长连接

### Q: 这个项目你做了什么？遇到的最大困难是什么？
> **我做的**：主导后端云函数架构设计，实现混合推荐算法和配送匹配算法，搭建 Docker 开发环境和 CI/CD 流水线，编写 44 个测试用例。
>
> **最大困难**：抢单并发控制。最初用 `doc.update` 直接更新，测试发现大量超卖。后来改成 `where + update` 原子操作才解决。这个坑让我深刻理解了 NoSQL 数据库的并发模型和一致性问题。

### Q: 线上如果出现问题怎么排查？
> 1. **日志**：云函数有 console 日志，搜索 error 级别
> 2. **监控**：Prometheus 面板看 FPS、API 延迟、错误率
> 3. **错误上报**：`reportError` 云函数记录客户端错误到 `error_logs` 集合
> 4. **请求追踪**：每个请求有 requestId，可以串联调用链
