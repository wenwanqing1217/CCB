# 面试话术手册

## 一、项目介绍（1分钟内）

**万能开场：**
> 这是我基于微信小程序云开发实现的校园盲盒即时配送平台，支持**C2C（个人对个人）**和**B2C（商家对个人）**双模式。个人可以发布闲置物品，周边商家可以入驻开店，学生骑手抢单配送，形成完整的校园交易闭环。

**技术栈一句话：**
> 微信小程序 + 云开发（云函数/云数据库/云存储）+ Node.js

---

## 二、商业模式讲解

### C2C模式（个人闲置交易）
**口述要点：**
> 学生可以把自己不需要的闲置物品以"盲盒"形式发布，设置价格和取货地点。买家下单后，骑手抢单并负责配送，帮助个人快速变现闲置。

### B2C模式（商家入驻）
**口述要点：**
> 校园周边的商家（如奶茶店、文具店、水果店等）可以申请认证入驻，在平台上发布商品，利用平台的校园流量和骑手网络，降低推广和配送成本。

## 三、核心功能讲解

### 1. 盲盒交易流程

**口述要点：**
> 用户可以把自己不需要的闲置物品以"盲盒"形式发布，设置价格和取货地点。买家下单后，骑手抢单并负责配送，形成闭环。

**面试追问——为什么用盲盒形式？**
> 盲盒形式降低了买卖双方的信息不对称，卖家不需要详细描述商品，买家有一定惊喜感，同时也能处理一些价值难以评估的闲置物品。

### 2. 即时配送系统

**口述要点：**
> 骑手在骑手端可以实时查看附近订单、抢单、开始配送。配送过程中骑手位置实时共享，买家可以查看骑手实时位置和预计到达时间。

**面试追问——配送员怎么知道订单？**
> 订单创建后进入待抢单池，骑手端会展示附近可抢订单列表，骑手根据距离、配送费等因素选择订单。使用了原子性更新防止多个骑手同时抢到同一订单。

### 3. AI智能助手与推荐系统

**口述要点：**
> 项目已经接入豆包大模型 API，实现了真正的智能问答。用户可以跟 AI 助手聊天，AI 会根据上下文智能回答，还能直接引导用户跳转到相应功能页面（如查看订单、发布盲盒等）。

**口述要点：**
> 我实现了混合推荐算法，结合内容推荐（根据商品特征匹配用户偏好）和协同过滤（根据相似用户行为推荐），按6:4权重混合，提升推荐准确率。

**面试追问——具体怎么实现的？**
> 内容推荐部分，我提取商品的分类、价格、标签等特征向量，与用户的兴趣画像计算余弦相似度。协同过滤部分，基于用户行为数据找相似用户，推荐相似用户喜欢的商品。最后按权重合并去重。

**面试追问——AI怎么实现的？**
> 我在云函数 aiService 中接入了豆包大模型 API，使用 Prompts 模板管理不同场景的对话（客服、推荐、润色等）。前端调用云函数获取 AI 回复，还实现了流式响应和对话上下文管理。

**面试追问——AI有做测试吗？**
> 有的！我专门做了 AI 专项评测，写了一个评测脚本（tests/ai_evaluator.py），包括基础功能、回答质量、性能压测、多轮对话等维度的测试，还生成了评测报告。

### 4. 订单状态实时同步

**口述要点：**
> 订单状态变更后，系统通过微信订阅消息推送给用户。同时骑手端和买家端都能看到实时状态更新。

**面试追问——怎么实现实时更新？**
> 有两种方案。方案一是前端轮询，每隔几秒请求一次云函数。方案二是云数据库实时监听，用watch()监听订单文档变更。方案二更高效，但我项目里主要用的是订阅消息推送+手动刷新结合。

## 四、技术难点与解决

### 1. 抢单并发问题

**口述：**
> 开发中遇到的最大问题是订单并发超卖——多个骑手同时抢单，可能都显示成功。解决方案是使用云数据库的原子性更新操作，在更新订单时先检查状态，只有状态为pending时才更新，通过affectedDocs判断是否更新成功。

```javascript
// 原子性更新示例
const updateResult = await ordersCollection.doc(orderId).update({
  data: { status: 'grabbed', riderOpenid }
})
if (updateResult.stats.updated === 0) {
  return { success: false, message: '订单已被抢' }
}
```

### 2. 推荐冷启动问题

**口述：**
> 新用户没有行为数据，推荐不准确。解决方案是新用户返回热门商品和新品，同时引导用户完善兴趣标签，逐步建立用户画像。

### 3. 分布式事务问题

**口述：**
> 创建订单涉及盲盒状态更新、订单插入等多个操作。云开发数据库支持单文档原子性，但对于跨文档操作，我通过合理的状态机设计保证最终一致性——比如先更新订单状态，再更新盲盒状态，如果中间失败有补偿机制。

### 4. 测试体系

**口述：**
> 我为项目搭建了完整的测试体系，包括50+详细测试用例、接口自动化测试、AI专项评测和并发测试。

**面试追问——测试都测了什么？**
> 首先梳理了50+测试用例（docs/测试用例.md），覆盖用户、盲盒、订单、配送、AI、管理等模块，包含正常、异常、边界、并发场景。
> 
> 然后用 Python + pytest 搭建了自动化测试框架，封装了各微服务的 API 接口。
> 
> 针对 AI 服务做了专项评测（tests/ai_evaluator.py），包括基础功能、回答质量、性能压测、多轮对话。
> 
> 针对抢单和下单做了并发测试，验证防超卖机制，确保只有一个骑手能抢到单，库存精确扣减。

## 五、项目亮点（说2-3个）

1. **C2C+B2C双模式**：既支持个人闲置交易，又支持商家入驻
2. **混合推荐算法**：内容+协同过滤，60:40权重混合
3. **原子性并发控制**：防止抢单超卖
4. **完整RBAC权限体系**：用户/商家/骑手/管理员四级权限
5. **完整微服务架构**：Spring Boot + Go + Python 多语言微服务
6. **完整测试体系**：50+测试用例、接口自动化、AI专项评测、并发测试
7. **AI大模型集成**：豆包API接入、智能客服、个性化推荐

## 六、不足与反思

**主动说：**
> 项目目前的不足是风控机制还不够完善，如果后续迭代，我会加入更严格的用户信用分和订单审核规则。另外AI功能目前是关键词匹配模拟，接入大模型可以大幅提升智能程度。

**面试追问——你觉得项目能商业化吗？**
> 核心功能已经具备，但商业化还需要：1）支付功能正式开通 2）风控体系完善 3）用户协议和法律合规 4）服务器性能和扩容方案。作为毕设项目，验证了技术可行性，但离商业化还有距离。
