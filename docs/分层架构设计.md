### 1.2 分层职责

| 层级 | 职责 | 关键规则 |
|------|------|---------|
| **接入层** | 小程序、页面渲染、用户交互 | 不包含业务逻辑、不暴露密钥 |
| **API网关层** | 路由转发、鉴权、限流、日志 | 统一入口、隐藏内部服务 |
| **应用服务层** | 业务逻辑、服务编排、参数校验 | 核心业务逻辑在这里 |
| **AI能力层** | 大模型调用、AI算法、推荐系统 | 纯AI逻辑，与业务解耦 |
| **数据层** | 数据读写、持久化、缓存、消息队列 | 下层不直接调用上层 |
| **监控运维层** | 指标采集、日志分析、链路追踪 | 全链路可观测 |

## 二、技术栈选型

### 2.1 选型表

| 层级 | 技术/组件 | 选型原因 |
|------|----------|---------|
| **接入层** | 微信小程序 (WXML/WXSS/JS) | 校园用户主要使用微信、开发成本低 |
| **API网关** | Nginx / Spring Cloud Gateway | 成熟稳定、社区活跃 |
| **应用服务层** | <br>- box-service: Spring Boot<br>- order-service: Spring Boot<br>- delivery-service: Go<br>- recommendation-service: Python | <br>- Spring Boot: 生态完善、业务开发快<br>- Go: 高并发、低延迟，适合位置服务<br>- Python: AI/推荐算法库丰富 |
| **AI能力层** | 火山方舟-豆包大模型 (预留) | 国内稳定、中文能力强、免费额度充足 |
| **数据层** | <br>- MySQL (主从)<br>- Redis (集群 + Geo)<br>- MongoDB<br>- RocketMQ<br>- Elasticsearch | <br>- MySQL: 结构化数据、成熟稳定<br>- Redis: 高性能缓存、Geo位置查询<br>- MongoDB: 文档存储、灵活Schema<br>- RocketMQ: 高吞吐、消息队列<br>- Elasticsearch: 搜索、推荐 |
| **监控运维** | Prometheus + Grafana + ELK + SkyWalking | 全链路可观测、生态完善 |
| **部署方案** | Docker + Docker Compose (本地) + Kubernetes (生产) | 环境统一、易扩展、易维护 |

### 2.2 选型三原则

| 原则 | 说明 |
|------|------|
| **适配原则** | 优先复用已有技术栈（云开发→微服务），降低成本 |
| **成熟原则** | 不追新，选主流稳定技术，避开小众框架 |
| **扩展原则** | 接口/模块设计预留扩展位（比如换模型、新增AI能力不用大幅改代码） |

## 三、AI应用专属设计

### 3.1 模型解耦

所有大模型调用逻辑封装成独立工具类，模型配置集中管理：

```
ai-service/
├── config/
│   └── model-config.yml    # 模型配置（名称、API地址、密钥）
├── services/
│   └── llm-service.js      # 统一LLM调用接口
└── prompts/                # Prompt配置文件
    ├── chat.prompt
    └── recommend.prompt
```

好处：未来换豆包→DeepSeek，只改配置不动业务代码。

### 3.2 Prompt 管理

所有提示词不硬写在业务代码，统一放到配置文件：

```yaml
# config/prompts.yml
chat:
  system_prompt: |
    你是校园盲盒平台的AI客服，友好、专业地解答用户问题。
    不要编造信息，如果不知道，引导用户联系人工客服。
recommend:
  system_prompt: |
    你是推荐助手，根据用户浏览历史推荐合适的盲盒。
```

### 3.3 Token 管控

- 上下文长度限制：只保留最近10轮对话
- 长对话自动截断：超出阈值时自动删除早期消息
- 请求频率限制：单用户每分钟最多5次请求

### 3.4 调用日志

关键AI请求、返回结果做日志记录：

```
日志内容：
- 用户ID
- 提问内容
- 模型版本
- 返回结果
- 耗时
- Token用量
```

## 四、非功能性设计

### 4.1 安全设计

| 措施 | 说明 |
|------|------|
| 密钥管理 | 密钥/鉴权只存服务端/环境变量，前端绝不暴露 |
| 接口鉴权 | 所有接口需Token验证，防非法调用 |
| 防刷机制 | 单用户请求频率限制、防重复请求 |
| 数据加密 | 敏感信息（手机号、地址）加密存储 |

### 4.2 异常容错

| 场景 | 处理方式 |
|------|---------|
| AI接口超时 | 重试2次，失败返回友好提示 |
| 网络异常 | 降级处理，返回缓存数据 |
| 库存不足 | 原子检查，友好提示"已售罄" |
| 抢单超卖 | 原子更新，通过affectedDocs判断 |

### 4.3 性能设计

| 优化点 | 方案 |
|-------|------|
| 高频数据 | Redis缓存（盲盒详情、热门推荐） |
| 位置查询 | Redis Geo，O(logN)复杂度 |
| 静态资源 | CDN加速 |
| 大列表 | 分页加载、虚拟滚动 |
| 异步处理 | 消息队列（订单创建→通知→配送） |

## 六、接口规范（示例）

### 6.1 接口文档模板

| 字段 | 说明 |
|------|------|
| 接口地址 | `/api/boxes` |
| 请求方式 | `GET` |
| 请求参数 | `page`, `size`, `category` |
| 返回字段 | `success`, `data`, `total`, `error` |
| 错误码 | `400`参数错误、`500`服务器错误 |
| 调用示例 | `GET /api/boxes?page=1&size=20&category=electronics` |

### 6.2 命名规范

- URL路径：小写、连字符分隔，如 `/api/user-orders`
- 函数/变量：驼峰命名，如 `getUserOrders`
- 常量：全大写下划线，如 `MAX_PAGE_SIZE=100`

## 七、Git分支规范

| 分支 | 说明 |
|------|------|
| `main` | 主分支，稳定版本，可部署 |
| `dev` | 开发分支，日常开发在此分支 |
| `feature/xxx` | 功能分支，新功能开发 |
| `hotfix/xxx` | 热修复分支，紧急Bug修复 |

### 提交规范

```
格式：<type>(<scope>): <subject>
type: feat/fix/docs/style/refactor/test/chore
示例：
- feat(box-service): 添加商品库存扣减功能
- fix(createOrder): 修复并发超卖问题
- docs(readme): 更新README文档
```

## 八、部署流程

### 本地开发环境

1. 启动Docker Compose：`docker-compose up -d`
2. 启动微服务：按需要分别启动
3. 本地验证：`http://localhost:8080`

### 生产环境

1. Git提交到main分支
2. CI/CD自动构建Docker镜像
3. K8s滚动更新部署
4. 监控告警验证
