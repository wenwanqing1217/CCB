# 校园盲盒平台 - 架构升级设计文档

## 一、现状分析

### 项目定位
校园场景下的**C2C+B2C**双模式盲盒交易与即时配送平台
- **C2C**：个人发布闲置物品盲盒
- **B2C**：认证商家发布商品

### 当前架构（云开发MVP）
```
微信小程序客户端
    ↓
微信云开发
    ├── 云函数（单体）
    ├── 云数据库（MongoDB）
    └── 云存储
```

### 当前问题
| 问题 | 说明 |
|------|------|
| 耦合度高 | 所有功能都在一个云开发环境里 |
| 扩容困难 | 云函数无法按服务独立扩容 |
| 性能瓶颈 | 无缓存层，每次都查数据库 |
| 响应慢 | 同步操作多，无消息队列解耦 |

## 四、微服务拆分方案

### 服务划分

| 服务名 | 端口 | 技术栈 | 职责（C2C+B2C支持） | 数据库 |
|--------|------|--------|--------------------|--------|
| user-service | 8001 | Spring Boot | 用户注册/登录/鉴权/RBAC<br>（支持个人用户、商家、骑手、管理员） | MySQL |
| box-service | 8002 | Spring Boot | 盲盒发布/查询/管理<br>（C2C：个人发布；B2C：商家发布、库存管理） | MySQL + MongoDB |
| order-service | 8003 | Spring Boot | 订单创建/支付/状态流转<br>（统一处理C2C和B2C订单） | MySQL |
| delivery-service | 8004 | Go | 骑手抢单/位置追踪/调度<br>（统一配送C2C和B2C订单） | Redis + MySQL |
| recommendation-service | 8005 | Python | 混合推荐算法/用户画像<br>（C2C和B2C商品混合推荐） | Redis + Elasticsearch |
| message-service | 8006 | Spring Boot | 订阅消息推送/站内信 | MySQL + RocketMQ |

### 服务间通信

| 场景 | 方式 | 说明 |
|------|------|------|
| 同步调用 | REST / gRPC | 订单服务调用用户服务查用户信息 |
| 异步通知 | RocketMQ | 订单创建成功 → 通知消息服务推送 |

## 五、缓存层设计

详细内容参考：[缓存层设计.md](./缓存层设计.md)

### 缓存策略

| 数据 | 缓存介质 | 过期时间 | 更新策略 |
|------|----------|----------|----------|
| 盲盒详情 | Redis LRU | 30分钟 | 主动更新 |
| 热门推荐 | Redis List | 5分钟 | 定时刷新 |
| 用户Token | Redis String | 7天 | 主动删除 |
| 骑手位置 | Redis Geo | 5秒 | 实时更新 |
| 附近订单 | Redis Geo | 1分钟 | 定时刷新 |

### 缓存一致性

采用 **Cache-Aside 模式**：
1. 读：先读缓存 → 命中返回 → 未命中读DB → 写缓存
2. 写：先写DB → 删缓存

### 缓存穿透/击穿/雪崩

| 问题 | 解决方案 |
|------|----------|
| 穿透 | 布隆过滤器 |
| 击穿 | 互斥锁 |
| 雪崩 | 随机过期时间 + 降级 |
