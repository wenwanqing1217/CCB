## 一、核心结论

**两种场景，两种做法：**

| 场景 | 开发顺序 | 适用情况 |
|------|---------|---------|
| **团队协作** | 先后端 → 再联调 → 最后前端 | 公司正式项目、多人团队、中大型系统 |
| **个人开发** | 设计阶段 → AI链路优先 → 前端并行 → 联调 → 迭代 → 收尾 | 你现在的主流场景 |

---

## 二、团队协作流程（标准工业流程）

### 2.1 标准顺序

```
1. 后端先行
   ├── 接口设计
   ├── 数据表设计
   ├── 服务开发
   ├── 自测
   └── 输出接口文档
           ↓
2. 前后端并行（提效）
   ├── 后端：继续完善其他接口
   └── 前端：基于接口文档 + Mock数据 并行开发
           ↓
3. 联调收尾
   ├── 对接真实接口
   ├── 联调Bug修复
   └── 异常处理
```

### 2.2 适用场景

- ✅ 公司正式项目
- ✅ 多人团队协作
- ✅ 中大型系统
- ✅ AI服务、支付、订单等核心模块

### 2.3 优点

| 优点 | 说明 |
|------|------|
| 接口定义统一 | 前后端不会各写一套逻辑 |
| 数据结构统一 | 后端统一把控业务规则 |
| 减少返工 | 避免前端写死数据结构，后端被迫迁就 |

## 四、特殊折中玩法

### 4.1 不推荐的玩法：页面先行

```
❌ 先做前端页面 → 后写接口
```

**弊端**：
- 前端容易写死数据结构
- 后端被迫迁就前端
- 架构混乱，后期难维护
- 违背分层解耦思想

**适用**：只适合纯展示、简单表单、原型验证

### 4.2 最推荐的玩法：接口先行 + Mock 并行

```
✅ 先定义接口文档 → 后端实现接口 → 前端用Mock数据 → 最后联调
```

**优点**：
- 和团队流程对齐
- 培养架构规范
- 效率和质量兼顾

## 五、四条关键原则（架构思维落地）

### 5.1 原则一：接口与数据结构，永远先于代码

```
✅ 先定契约（接口、字段、状态）
❌ 不要先写代码再补接口
```

**操作方法**：
1. 开发任何功能前，先写接口文档
2. 包括：地址、入参、出参、错误码
3. 数据结构、枚举值提前定义

### 5.2 原则二：外部依赖优先开发测试

```
✅ AI大模型、第三方接口、支付等外部服务，优先打通
❌ 不要最后再做外部依赖，风险后置
```

**原因**：
- 外部服务不稳定，风险最高
- 越早发现依赖问题，修复成本越低
- 避免项目收尾时才发现外部依赖不可用

### 5.3 原则三：核心链路优先，边缘功能后置

```
✅ 先保证主干流程能跑
❌ 不要先做美化、拓展功能
```

**操作方法**：
1. 先跑通「下单→支付→配送→完成」核心链路
2. 再做积分、收藏、社区等边缘功能
3. 再做优化、美化

### 5.4 原则四：拒绝"页面写死再补接口"

```
❌ 页面写死 → 再补接口 → 后端被迫迁就 → 架构腐化
✅ 接口先行 → 前端用Mock → 后端实现 → 联调 → 架构清晰
```

**长期价值**：
- 培养架构思维
- 方便后续迭代
- 利于团队协作

```javascript
function aiChat(question) {
```

#### 改进方案：先打通豆包API


#### 改进后的 aiChat 云函数

```javascript
// 云函数入口
const axios = require('axios');

// 豆包API配置
const CONFIG = {
  apiKey: process.env.DOUBAO_API_KEY,  // 环境变量，不暴露
  endpoint: 'https://ark.cn-beijing.volces.com/api/v3/chat/completions'
};

// 统一AI调用接口
async function callAI(messages) {
  try {
    const response = await axios.post(CONFIG.endpoint, {
      model: 'doubao-pro',
      messages: messages,
      max_tokens: 500,
      temperature: 0.7
    }, {
      headers: {
        'Authorization': `Bearer ${CONFIG.apiKey}`,
        'Content-Type': 'application/json'
      },
      timeout: 10000  // 10秒超时
    });
    
    return {
      success: true,
      data: response.data.choices[0].message.content
    };
  } catch (error) {
    // 统一错误处理
    return {
      success: false,
      error: error.message
    };
  }
}

// 云函数入口
exports.main = async (event, context) => {
  const { question, history = [] } = event;
  
  // 构建消息格式
  const systemPrompt = `你是校园盲盒平台的AI客服，友好、专业地解答用户问题。
不要编造信息，如果不知道，引导用户联系人工客服。`;
  
  const messages = [
    { role: 'system', content: systemPrompt },
    ...history,
    { role: 'user', content: question }
  ];
  
  // 调用AI
  const result = await callAI(messages);
  
  return result;
};
```

### 6.3 接口文档模板

#### AI对话接口

| 字段 | 值 |
|------|-----|
| 接口地址 | `/api/ai/chat` |
| 请求方式 | `POST` |
| 鉴权要求 | 是（用户Token） |

**请求参数**：

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| question | string | 是 | 用户问题 |
| history | array | 否 | 对话历史[[role, content], ...] |

**返回字段**：

| 字段 | 类型 | 说明 |
|------|------|------|
| success | boolean | 是否成功 |
| data | string | AI回答内容 |
| error | string | 错误信息（失败时） |

**调用示例**：

```javascript
// 前端调用
wx.cloud.callFunction({
  name: 'aiChat',
  data: {
    question: '这个盲盒什么时候能送到？',
    history: [
      ['user', '你好'],
      ['assistant', '您好！有什么可以帮助您的？']
    ]
  }
}).then(res => {
  if (res.result.success) {
    this.setData({ answer: res.result.data });
  } else {
    this.setData({ answer: '抱歉，AI服务暂时不可用' });
  }
});
```

## 七、面试回答模板

### 问：前后端开发顺序是怎样的？

**团队协作版**：
> 团队协作中我遵循**先定义接口与数据模型，后端优先开发并输出接口文档，前端基于文档和Mock数据并行开发，最后统一联调**的流程。

**个人开发版（AI类项目）**：
> 个人独立开发AI类项目时，会**优先打通大模型调用这条核心链路**，再开发前端页面，保证核心功能稳定，同时全程遵守分层、解耦的设计规范。

## 八、养成规范习惯检查清单

每次开发新功能前，检查：

- [ ] **接口文档先写**：接口地址、入参、出参、错误码
- [ ] **数据结构先定**：数据表、枚举值、字段类型
- [ ] **AI链路优先**：如果是AI功能，先打通大模型
- [ ] **前端Mock并行**：前端用假数据先开发，不用等后端
- [ ] **联调留时间**：不要最后一天才联调
- [ ] **核心链路先跑通**：先跑通主流程，再做边缘功能
