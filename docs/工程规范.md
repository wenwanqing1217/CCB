## 一、编码规范

### 1.1 命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 文件夹 | 小写、连字符分隔 | `box-service`, `user-profile` |
| 文件 | 小写、连字符分隔 | `order-controller.js`, `api-helper.go` |
| 变量/函数 | 小驼峰 | `getUserOrders`, `calculatePrice` |
| 常量 | 全大写下划线 | `MAX_PAGE_SIZE`, `DEFAULT_EXPIRE_TIME` |
| 类名 | 大驼峰 | `BoxController`, `OrderService` |
| CSS类 | 小写、连字符 | `item-card`, `btn-primary` |

### 1.2 注释规范

#### 函数注释（JSDoc风格）

```javascript
/**
 * 创建订单
 * @param {string} userId - 用户ID
 * @param {string} boxId - 盲盒ID
 * @param {string} address - 收货地址
 * @returns {Promise<Order>} 创建的订单对象
 * @throws {Error} 库存不足时抛出异常
 */
async function createOrder(userId, boxId, address) {
  // ...
}
```

#### 复杂业务逻辑注释

```javascript
// 原子性抢单：先检查订单状态，再更新，通过affectedDocs判断是否成功
// 防止多个骑手同时抢到同一订单
const updateResult = await db.collection('orders').doc(orderId).update({
  data: {
    status: 'grabbed',
    riderId: riderId,
    updateTime: new Date()
  }
});

if (updateResult.stats.updated === 0) {
  throw new Error('订单已被抢');
}
```

### 1.3 代码拆分原则

- **通用逻辑抽离**：工具函数、公共组件不要重复写
- **单一职责**：一个函数只做一件事
- **避免深层嵌套**：超过3层嵌套考虑重构

## 三、Git规范

### 3.1 分支管理

| 分支 | 说明 | 使用场景 |
|------|------|---------|
| `main` | 主分支，稳定版本 | 生产环境、发版 |
| `dev` | 开发分支 | 日常开发、集成测试 |
| `feature/功能名称` | 功能分支 | 新功能开发 |
| `hotfix/修复内容` | 热修复分支 | 紧急Bug修复 |

### 3.2 分支流程

```
main
└── dev
    ├── feature/add-ai-chat  → 开发完合并回dev
    └── hotfix/fix-order-bug → 开发完合并回main和dev
```

### 3.3 提交信息规范

```
格式：<type>(<scope>): <subject>

type类型：
- feat: 新功能
- fix: 修复Bug
- docs: 文档更新
- style: 代码格式调整
- refactor: 重构
- test: 测试相关
- chore: 构建/工具/依赖

示例：
- feat(box-service): 添加商品库存扣减功能
- fix(createOrder): 修复并发超卖问题
- docs(readme): 更新README文档
- refactor(recommend-service): 重构推荐算法
```

### 3.4 提交原则

- 一次提交只做一件事
- 提交前确保代码可编译、可运行
- 重要版本打标签：`v1.0.0`, `v1.1.0`

## 四、接口文档规范

### 4.1 接口文档模板

#### 接口信息

| 字段 | 值 |
|------|-----|
| 接口地址 | `/api/boxes` |
| 请求方式 | `GET` |
| 鉴权要求 | 否 |

#### 请求参数

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| page | int | 否 | 页码，默认1 |
| size | int | 否 | 每页数量，默认20 |
| category | string | 否 | 分类筛选 |

#### 返回字段

| 字段 | 类型 | 说明 |
|------|------|------|
| success | boolean | 是否成功 |
| data | array | 盲盒列表 |
| total | number | 总数量 |
| error | string | 错误信息（失败时） |

#### 错误码

| 错误码 | 说明 |
|--------|------|
| 200 | 成功 |
| 400 | 参数错误 |
| 500 | 服务器错误 |

#### 调用示例

```bash
# 请求示例
curl -X GET "http://localhost:8002/api/boxes?page=1&size=20"

# 响应示例
{
  "success": true,
  "data": [...],
  "total": 100
}
```

## 五、配置规范

### 5.1 配置文件分类

| 配置文件 | 说明 |
|---------|------|
| `application.yml` | 基础配置 |
| `application-dev.yml` | 开发环境配置 |
| `application-prod.yml` | 生产环境配置 |
| `.env` | 环境变量（密钥、敏感信息，不提交Git） |

### 5.2 敏感信息管理

- ❌ 不要硬编码密钥、密码
- ✅ 使用环境变量或配置中心
- ✅ `.env` 文件加入 `.gitignore`

```yaml
# application.yml (示例)
spring:
  datasource:
    url: ${DB_URL:jdbc:mysql://localhost:3306/box}
    username: ${DB_USERNAME:root}
    password: ${DB_PASSWORD}  # 环境变量注入
```

## 六、测试规范

### 6.1 测试层级

| 层级 | 说明 | 覆盖率目标 |
|------|------|-----------|
| 单元测试 | 单个函数/类的测试 | 核心逻辑>80% |
| 集成测试 | 模块间交互的测试 | 关键流程全覆盖 |
| E2E测试 | 完整用户流程的测试 | 核心流程全覆盖 |

### 6.2 测试原则

- 先写测试，再写代码（TDD可选）
- 写完功能立即写测试
- 提交代码前确保测试通过
