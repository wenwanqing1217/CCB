# 分库分表设计文档

## 一、背景

随着业务发展，数据量快速增长：
- 订单表：预计月增 100 万条
- 用户表：预计 100 万用户
- 配送记录表：预计月增 500 万条

单库单表性能瓶颈：
- 查询变慢
- 写入变慢
- 扩容困难

---

## 二、分库分表方案

### 分库策略

| 库名 | 分片键 | 分片规则 | 库数量 |
|------|--------|----------|--------|
| user_db | user_id | user_id % 4 | 4 |
| order_db | create_time | YYYYMM | 12/年 |
| box_db | box_id | box_id % 4 | 4 |
| delivery_db | rider_id | rider_id % 4 | 4 |

### 分表策略

| 表名 | 分片键 | 分片规则 | 表数量/库 |
|------|--------|----------|-----------|
| users | user_id | user_id % 16 | 16 |
| orders | create_time | YYYYMM | 12 |
| order_logs | create_time | YYYYMMDD | 365 |
| boxes | box_id | box_id % 16 | 16 |
| delivery_records | rider_id | rider_id % 16 | 16 |

**路由规则：**
```java
public String getUserDataSource(String userId) {
    long id = Long.parseLong(userId);
    int dbIndex = (int) (id % 4);
    return "user_db_" + dbIndex;

    int tableIndex = (int) (id % 16);
    return "users_" + tableIndex;
```

## 五、跨分片查询解决方案

### 场景1：查询某个用户的所有订单

**问题：** 订单表按时间分表，一个用户的订单可能在多个表

**解决方案：**
1. 在用户表冗余最近3个月的订单ID列表
2. 使用 Elasticsearch 做订单搜索
3. 限制查询时间范围（只查最近3个月）

### 场景2：统计全站订单数

**问题：** 订单表分散在多个库，统计需要查所有表

**解决方案：**
1. 定时任务汇总统计数据到 Redis
2. 使用数据仓库（ClickHouse/MaxCompute）做离线分析
3. 实时计算（Flink）

## 六、分布式主键

### 方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| UUID | 简单 | 无序，索引性能差 |
| 数据库自增 | 简单 | 单点瓶颈，分库分表困难 |
| Snowflake | 有序，高性能 | 依赖时钟 |
| Redis 自增 | 高性能 | 依赖 Redis |

**推荐：** Snowflake

```java
public class SnowflakeIdGenerator {
    
    private final long epoch = 1690000000000L; // 2023-07-22
    private final long workerIdBits = 5L;
    private final long dataCenterIdBits = 5L;
    private final long sequenceBits = 12L;
    
    private final long workerId;
    private final long dataCenterId;
    private long sequence = 0L;
    private long lastTimestamp = -1L;
    
    public SnowflakeIdGenerator(long workerId, long dataCenterId) {
        this.workerId = workerId;
        this.dataCenterId = dataCenterId;
    }
    
    public synchronized long nextId() {
        long timestamp = System.currentTimeMillis();
        
        if (timestamp < lastTimestamp) {
            throw new RuntimeException("时钟回拨");
        }
        
        if (timestamp == lastTimestamp) {
            sequence = (sequence + 1) & 0xFFF;
            if (sequence == 0) {
                timestamp = tilNextMillis(lastTimestamp);
            }
        } else {
            sequence = 0L;
        }
        
        lastTimestamp = timestamp;
        
        return ((timestamp - epoch) << (workerIdBits + dataCenterIdBits + sequenceBits))
                | (dataCenterId << (workerIdBits + sequenceBits))
                | (workerId << sequenceBits)
                | sequence;
    }
    
    private long tilNextMillis(long lastTimestamp) {
        long timestamp = System.currentTimeMillis();
        while (timestamp <= lastTimestamp) {
            timestamp = System.currentTimeMillis();
        }
        return timestamp;
    }
}
```

## 七、扩容方案

### 垂直扩容（升级配置）
- 暂时方案
- 快速但有上限

### 水平扩容（加机器）

**方案：** 双写 + 迁移
```
1. 新写入同时写旧库和新库（双写）
2. 后台程序迁移旧数据到新库
3. 数据校验一致后，读流量切到新库
4. 停止双写，只写新库
```

## 八、常见问题

### 问题1：分片键选择

**原则：**
1. 查询频率最高的字段
2. 数据分布均匀
3. 不会变化

**推荐：**
- 用户表：user_id
- 订单表：create_time（或 order_id）
- 配送表：rider_id

### 问题2：分片数据倾斜

**原因：** 某些分片数据太多，某些太少

**解决方案：**
1. 分析数据分布，调整分片算法
2. 使用一致性哈希
3. 读写分离

### 问题3：事务一致性

**方案：**
1. 本地事务（同一分片内）
2. 分布式事务（Seata）
3. 最终一致性（消息队列）
