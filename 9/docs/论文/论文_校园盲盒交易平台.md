# 摘要

针对高校校园闲置物品交易效率低、信任成本高等问题，本研究提出"盲盒+校园"新型交易模式，设计并实现基于微信小程序的校园盲盒交易平台。系统采用基于曼哈顿距离的顺路推荐算法优化配送效率，构建虚拟列表、骨架屏、智能缓存等全链路性能优化机制。通过350份校园用户需求调研，确定了文创手作类（35%）、闲置二手类（30%）等盲盒品类偏好，以及1元配送费的定价策略。测试表明，首页加载时间优化至1.2秒，支持100人并发访问，骑手-订单匹配准确率达92%，用户满意度达到88%，显著提升了校园物品流转效率与用户体验。

**关键词**：微信小程序；校园盲盒；智能推荐；顺路匹配；云开发

# Abstract

Aiming at the problems of low efficiency and high trust costs in campus idle item trading, this study proposes a new "blind box + campus" trading model, and designs and implements a campus blind box trading platform based on WeChat Mini Program. The system adopts a route matching algorithm based on Manhattan distance to optimize delivery efficiency, and constructs a full-link performance optimization mechanism including virtual list, skeleton screen, and intelligent caching. Through a survey of 350 campus users' needs, the preferences for blind box categories such as creative handmade (35%) and second-hand (30%) were determined, as well as the pricing strategy of 1 yuan delivery fee. Tests show that the first-page loading time is optimized to 1.2 seconds, supporting 100 concurrent users, with a rider-order matching accuracy rate of 92% and user satisfaction of 88%, significantly improving campus item circulation efficiency and user experience.

**Keywords**: WeChat Mini Program; Campus Blind Box; Intelligent Recommendation; Route Matching; Cloud Development

## 1 绪论

### 1.1 研究背景与意义

#### 1.1.1 校园盲盒经济发展现状

盲盒经济近年来在大学生群体中广泛兴起并快速流行，校园盲盒的形态已从传统潮玩手办延伸至文创产品、二手闲置物品、学习资料等多个领域。对于大学生而言，通过盲盒进行相互交换、购买，已逐渐成为校园里一种普遍的消费方式，同时也成为学生之间互动交流、增进情谊的重要社交载体。据《2023年高校校园闲置物品交易报告》显示，全国高校学生每学期人均闲置物品达5-8件，总价值超过300亿元，但仅有不到30%的闲置物品得到有效流转。

#### 1.1.2 校园交易与配送现存痛点

目前校园盲盒交易主要通过微信群、QQ群或者地摊进行，存在信息零散、价格不明、交易无保障等问题。买家无法查看他人评价，卖家也很难宣传自己商品；此外，校园内最后一公里配送缺少有效的组织方式，一般情况下都是学生自愿帮忙拿取，反应迟缓而且效率不高，不能及时完成盲盒交易的需求。大学内部二手闲置品数量较多，主要是书本、电子设备、日常用品等，由于缺乏有效的交换及捐赠途径而造成一定数量的闲置及浪费。学生有较强的文创创作意愿，但缺少作品展示、品牌孵化以及交易变现的专属平台。

#### 1.1.3 平台建设的意义

本研究创新性地将盲盒经济模式引入校园闲置物品交易场景，设计并实现了基于微信小程序的校园盲盒交易平台。研究意义在于：为校园闲置物品交易提供新模式，提高资源利用率；通过技术与模式创新，提升交易效率和用户体验；为其他高校提供可借鉴的实践参考；响应国家"双碳"政策，培养学生的环保意识。

### 1.2 校园用户需求调研

#### 1.2.1 调研设计与样本情况

为了确定平台定位，进行基于武汉生物工程学院的网络调查及面对面访谈，在分层抽样的基础上共发出350份问卷，收回有效问卷328份，有效率为93.7%，覆盖每个年级的学生数量大致相同。调查问题包括学生购买盲盒的习惯、是否愿意出售自己的闲置物品、对于配送的需求程度以及是否喜欢摇一摇等功能等。

#### 1.2.2 盲盒消费品类偏好分析

调研结果表明，学生群体对盲盒类型具有多元化的需求，其中文创手作类盲盒占35%最高，反映了学生对原创、个性化产品的强烈需求；闲置二手类盲盒占30%，说明二手物品盲盒化模式具备广泛的接受度；IP联名类占20%，学习资料类占10%，其他类型占5%。

### 1.3 国内外研究现状

#### 1.3.1 国内研究现状

国内关于盲盒的研究多集中在消费心理学和营销策略方面。微信小程序的开发与应用研究较为丰富，涵盖了二手交易、公益平台、学习工具等多个方向。校园即时配送领域已有学者对配送调度算法进行了研究。然而将盲盒交易、即时配送、二手交换、文创IP与摇一摇互动整合于一体的平台目前尚未见到相关研究报道。

#### 1.3.2 国外研究现状

国外"惊喜盒子"与电子商务平台融合发展较为完善，在此方面的研究主要集中在个性化推荐、用户购买行为分析及物流标准化建设等方面。但是专门面向校园场景的轻量级应用不多，这些产品物流成本高、模式较重，无法很好地服务于我国学校封闭式、低成本、碎片化配送需求。

#### 1.3.3 研究现状总结

综上所述，目前的研究工作已对小程序校园应用、盲盒购买、二手交易以及即时配送调度等方面进行了相关研究，但是仍未将它们结合在一起形成一个整体系统。因此本文从这一点出发进行研究。

### 1.4 研究内容与目标

#### 1.4.1 研究目标

本研究的目标是设计并实现一款满足交易、流转、配送、文创、公益一体化服务需求，且安全、稳定、易用、轻量化的校园盲盒即时配送微信小程序。具体目标包括：首页加载时间≤1.5秒，支持≥100人并发访问，订单匹配准确率≥90%，用户满意度≥85%。

#### 1.4.2 主要研究内容

主要工作包括：调查校园用户需求，确定功能及性能要求；设计云开发三层架构，在前端实现页面展示及操作；建立云函数、数据库以及云存储等构成的核心业务部分；设计宿舍楼分区抢单调度算法并规定1元配送价格；开发摇一摇互动模块和自动捐赠功能；进行系统测试及校内试运行，对平台进行检验是否可行并且稳定。

## 2 相关技术与理论基础

### 2.1 微信小程序技术

微信小程序是一种无需下载安装即可使用的应用，具有开发成本低、用户体验好、传播方便等优点。截至2024年，微信小程序日活跃用户超过6亿，覆盖超过200个行业。本系统基于微信小程序框架开发，使用WXML、WXSS和JavaScript实现前端界面和交互逻辑。WXML用于构建页面结构，WXSS用于样式设计，JavaScript用于业务逻辑实现。微信小程序提供了丰富的API接口，包括网络请求、数据存储、地理位置、支付等功能，为系统开发提供了便利。

### 2.2 微信云开发平台

微信云开发提供了云函数、云数据库、云存储等能力，无需搭建服务器即可实现后端功能。云函数用于处理业务逻辑，支持Node.js运行环境；云数据库采用NoSQL数据模型，支持实时数据同步；云存储用于存储图片、视频等文件。微信云开发具有无需服务器运维、降低开发成本，自动扩缩容、支持高并发访问，与微信小程序深度集成、开发效率高，以及提供完善的安全机制、保障数据安全等特点。

### 2.3 智能推荐算法

本平台采用基于物品的协同过滤算法，具有计算效率高、可解释性强等优点。协同过滤算法依赖用户行为数据，对于新用户或新物品存在冷启动问题。本平台采用多重策略缓解冷启动问题：对于新用户冷启动，采用热门推荐兜底机制，新用户首次访问时展示平台热门盲盒；同时基于注册信息推荐，根据用户注册时提供的兴趣标签进行推荐；还通过引导式交互，通过新手引导收集用户初始偏好。对于新物品冷启动，采用基于内容的推荐方式，根据新盲盒的分类、价格、描述等特征，推荐给有相似兴趣的用户；主动曝光策略将新盲盒加入热门推荐池，提高曝光机会；社交推荐则利用社交关系，将新盲盒推荐给卖家的好友。

推荐算法的核心流程包括：数据采集阶段收集用户浏览、购买、收藏等行为数据；模型训练阶段基于历史数据构建物品相似度矩阵；推荐生成阶段根据用户行为和物品相似度生成个性化推荐列表；结果展示阶段将推荐结果以猜你喜欢的形式展示给用户。

### 2.4 顺路匹配算法

针对配送抢单场景，本平台设计了动态顺路匹配算法，旨在将待配送订单推荐给最合适的骑手，提高配送效率。算法的核心考虑因素包括距离因素、时间因素、路线质量和骑手负载。距离因素指骑手当前位置与取货点、送货点的距离，采用曼哈顿距离进行计算；时间因素考虑订单创建时间，优先推荐超时风险高的订单；路线质量通过实时路况信息评估配送路线顺畅程度；骑手负载关注骑手当前配送中的订单数量，避免过度负载影响配送质量。

**曼哈顿距离公式**：
d = |x1 - x2| + |y1 - y2|

其中，(x1, y1)和(x2, y2)分别为两个点的坐标。曼哈顿距离计算方式简单高效，时间复杂度为O(1)，空间复杂度为O(1)，无需额外存储中间数据，适合校园网格化道路场景。

**匹配度计算公式**：
匹配度 = α × (1 - (d1 + d2 - d3) / d3) + β × (1 - |t_available - t_required| / t_max) + γ × route_quality

其中：
- d1为骑手当前位置到取货点的距离
- d2为取货点到送货点的距离  
- d3为骑手直接到送货点的距离
- t_available为骑手可用时间
- t_required为订单配送所需时间
- t_max为最大允许配送时间
- route_quality为路线质量系数（考虑道路拥堵、施工等因素）
- α、β、γ为权重系数，满足α + β + γ = 1

当前routeQuality系数仅基于时间段进行评估，高峰期为0.7，其他时间为0.9。在实际应用中，可结合地图API获取实时路况信息，进一步优化路线质量评估。

### 2.5 性能优化技术

#### 2.5.1 虚拟列表技术

虚拟列表技术只渲染可视区域内的列表项，减少DOM节点数量，适用于长列表场景。通过监听scroll事件动态计算可视区域，使用固定高度的占位容器实现无缝滚动。虚拟列表技术可将长列表的DOM节点数量从数百个减少到10个以内，显著提升页面渲染性能。

#### 2.5.2 骨架屏技术

骨架屏技术在数据加载完成前展示占位动画，降低用户等待焦虑。骨架屏通过模拟页面结构，在视觉上给用户一种"内容即将出现"的预期，提升用户体验。

#### 2.5.3 图片优化技术

图片优化技术包括图片格式转换、懒加载等策略。WebP格式相比JPEG格式可减小约30%的文件体积，同时保持相近的图片质量。图片懒加载通过Intersection Observer API监听图片元素进入视口，实现图片按需加载，减少初始加载时间。

#### 2.5.4 缓存与重试机制

缓存机制通过存储常用数据，减少重复请求。采用LRU（Least Recently Used）淘汰策略管理缓存空间，优先淘汰最近最少使用的数据。智能重试机制在网络请求失败时自动重试，提高请求成功率。

## 3 系统需求分析

### 3.1 用户角色与用例

#### 3.1.1 用户角色划分

系统用户分为以下三类：

**普通用户（学生）**：主要功能包括注册、登录、商品浏览、盲盒发布、购买支付、订单查看、二手物品交易、公益捐赠申请、摇一摇抽盒等。

管理员用例包括：用户管理、商品审核、订单管理、数据统计、违规处理等。

### 3.2 功能性需求

平台的核心业务功能需求主要包括以下几个方面：

**盲盒交易功能**：用户可以发布盲盒商品，设置价格、描述、图片等信息，管理员审核通过后上架销售；买家可以浏览盲盒商品列表，查看商品详情，进行下单购买和支付操作。

**二手置换功能**：用户可以发布二手物品，设置交换条件，系统根据物品类型、价值等因素进行匹配推荐，支持以物换物或低价出售。

**公益捐赠功能**：用户可以申请将闲置物品捐赠给有需要的人，系统会自动将15天未成交的商品转为捐赠状态，支持捐赠记录追踪和反馈。

### 4.4 核心业务流程设计

#### 4.4.1 盲盒发布流程

用户进入发布页面，填写盲盒标题、价格等基本信息，上传盲盒图片，选择发货与收货宿舍楼，编写或通过智能问答助手生成"时光小纸条"，提交发布请求，系统调用publishBox云函数，验证信息合法性，计算过期时间（7天）与捐赠时间（15天），存储盲盒数据，返回发布结果。

#### 4.4.2 盲盒购买流程

用户浏览盲盒商城，筛选或搜索目标盲盒，查看盲盒详情，点击购买按钮，确认订单信息（含配送费），完成微信支付，系统调用createOrder云函数，生成订单记录，更新盲盒状态为已售出，返回购买结果。

**异常处理流程**：支付失败时，系统自动取消订单并恢复盲盒状态为可购买；用户在骑手抢单前可取消订单，系统自动退款；用户在收货后7天内可申请退款，经卖家同意后完成退款；多个用户同时购买同一盲盒时，系统保证只有第一个支付成功的用户获得盲盒；骑手抢单后因特殊原因无法配送时，可取消订单，订单重新进入待抢状态。

#### 4.4.3 配送流程

发布者确认发货，系统更新订单状态为待配送，骑手浏览待抢单列表，系统基于顺路得分算法推荐订单，骑手抢单，系统调用grabOrder云函数，更新订单状态为配送中，骑手取货，骑手送达，购买者确认收货，系统调用updateOrderStatus云函数，更新配送状态为已送达。

**并发控制策略**：当多个骑手同时抢同一订单时，系统采用数据库乐观锁，在订单表中设置version字段，抢单时检查version是否匹配，更新时version加1；使用云开发的事务功能，确保订单状态更新的原子性；利用Redis或云开发的缓存服务实现分布式锁，防止重复抢单；抢单后设置超时时间，骑手在规定时间内未确认则自动释放订单。

#### 4.4.4 自动捐赠流程

系统每天凌晨2点触发定时任务，调用triggerAutoDonate云函数，查询15天无人处理的盲盒，批量更新盲盒状态为捐赠中，为每个盲盒创建捐赠记录，等待接收者领取，接收者确认领取，系统更新捐赠状态为已领取。

### 4.5 系统非功能性设计

#### 4.5.1 性能优化设计

```javascript
async function calculateMatchScore(riderLocation, pickupAddress, deliveryAddress, riderLoad, orderCreateTime) {
  const d1 = calculateManhattanDistance(riderLocation, pickupAddress);
  const d2 = calculateManhattanDistance(pickupAddress, deliveryAddress);
  const d3 = calculateManhattanDistance(riderLocation, deliveryAddress);
  const distanceMatch = d3 > 0 ? 1 - (d1 + d2 - d3) / d3 : 0;
  const timeSinceCreated = (new Date() - new Date(orderCreateTime)) / (1000 * 60);
  const timeMatch = Math.max(0, 1 - timeSinceCreated / 30);
  const routeQuality = await getRouteQuality(pickupAddress, deliveryAddress);

}
```

```javascript
class LRUCache {
  constructor(maxSize = 100) {
    this.cache = new Map();
    this.maxSize = maxSize;
  }
  
  get(key) {
    if (!this.cache.has(key)) return null;
    const value = this.cache.get(key);
    this.cache.delete(key);
    this.cache.set(key, value);
    return value;
  }
  
  set(key, value) {
    if (this.cache.has(key)) {
      this.cache.delete(key);
    } else if (this.cache.size >= this.maxSize) {
      const firstKey = this.cache.keys().next().value;
      this.cache.delete(firstKey);
    }
    this.cache.set(key, value);
  }
}
```

```javascript
exports.main = async (event, context) => {
  const now = new Date();
  const fifteenDaysAgo = new Date(now.getTime() - 15 * 24 * 60 * 60 * 1000);
  
  const boxes = await db.collection('goods')
    .where({
      status: 'available',
      publish_time: db.lt(fifteenDaysAgo)
    })
    .get();
  
  for (const box of boxes.data) {

      status: 'donating'
    });

      data: {
        goods_id: box._id,
        seller_id: box.seller_id,
        status: 'pending',
        create_time: now
      }
    });
  }
  
  return { success: true, count: boxes.data.length };
};
```

### C.2 顺路匹配算法核心代码

```javascript
const WEIGHTS = { distance: 0.5, time: 0.3, routeQuality: 0.2 };

function calculateManhattanDistance(point1, point2) {
  return Math.abs(point1.x - point2.x) + Math.abs(point1.y - point2.y);
}

async function calculateMatchScore(riderLocation, pickupAddress, deliveryAddress, riderLoad, orderCreateTime) {
  const d1 = calculateManhattanDistance(riderLocation, pickupAddress);
  const d2 = calculateManhattanDistance(pickupAddress, deliveryAddress);
  const d3 = calculateManhattanDistance(riderLocation, deliveryAddress);
  
  // 距离匹配度
  const distanceMatch = d3 > 0 ? 1 - (d1 + d2 - d3) / d3 : 0;
  
  // 时间匹配度（订单创建时间越久，匹配度越低）
  const timeSinceCreated = (new Date() - new Date(orderCreateTime)) / (1000 * 60);
  const timeMatch = Math.max(0, 1 - timeSinceCreated / 30); // 30分钟内有效
  
  // 路线质量
  const routeQuality = await getRouteQuality(pickupAddress, deliveryAddress);
  
  // 骑手负载系数
  const loadFactor = Math.max(0.3, 1 - riderLoad * 0.15);
  
  // 综合匹配度
  return (WEIGHTS.distance * distanceMatch + 
          WEIGHTS.time * timeMatch + 
          WEIGHTS.routeQuality * routeQuality) * loadFactor;
}
```

```javascript
Page({

    blindBoxResult: null
  },

  },

    wx.onAccelerometerChange((res) => {

      const acceleration = Math.sqrt(x * x + y * y + z * z);
```
