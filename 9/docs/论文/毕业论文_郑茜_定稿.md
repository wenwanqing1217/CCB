## 摘要

目前校园盲盒交易主要通过微信群或者学校内的小摊贩进行，缺少一个统一规范化的线上运营渠道，在校园盲盒交易中还存在校园末端配送效率较低、二手闲置物品流通困难等问题，为了解决这些问题，本文设计并实现了一个基于微信小程序云开发的校园盲盒即时配送以及互动平台。

该平台集成了盲盒交易、二手置换、公益捐赠、文创IP运营以及即时配送等五大功能模块，并加入了"摇一摇"随机抽取盲盒的功能，在前端使用紫色霓虹灯颜色作为整体色调并通过手机加速度传感器实现摇一摇功能，在校内盲盒配送上采用按宿舍楼划分区域抢夺订单的方式并且规定所有配送费均为1元。

经过一系列测试以及在学校内部进行一定时间实验后，平台页面平均加载时间为1.2秒，接口平均响应时间为300ms，订单抢单成功率高达93.5%，摇一摇随机盲盒游戏流畅性良好。整个平台各项功能完善并且能够正常运作，可以很好地实现盲盒交易、即时配送及摇一摇随机盲盒游戏等功能。

**关键词**：微信小程序；校园盲盒；即时配送；云开发；分区抢单

## Abstract

Currently, campus blind box transactions are mainly conducted through WeChat groups or on-campus vendors, lacking a unified and standardized online operation channel. Problems such as low efficiency of last-mile delivery on campus and difficulties in the circulation of second-hand idle items also exist in campus blind box trading. To address these issues, this paper designs and implements a campus blind box instant delivery and interactive platform based on WeChat Mini Program cloud development.

The platform integrates five functional modules: blind box trading, second-hand exchange, public welfare donation, cultural and creative IP operation, and instant delivery. It also incorporates a "shake-to-draw" function for random blind box selection. The front-end adopts a purple neon style as the overall tone and implements the shake function through the mobile phone accelerometer. For campus blind box delivery, a dormitory-based order-grabbing mechanism is adopted, with a unified delivery fee of 1 yuan for all orders.

After systematic testing and on-campus trial operation, the platform achieves an average page loading time of 1.2 seconds, an average interface response time of 300 ms, an order-grabbing success rate of 93.5%, and smooth operation of the "shake-to-draw" interactive game. The platform features complete functions and stable operation, effectively realizing blind box trading, instant delivery, and the random blind box interactive game.

**Keywords**：WeChat Mini Program; Campus Blind Box; Instant Delivery; Cloud Development; Zone-based Order-Grabbing

## 1 绪论

### 1.1 研究背景与意义

#### 1.1.1 校园盲盒经济发展现状

盲盒经济近年来在大学生群体中广泛兴起并快速流行，校园盲盒的形态已从传统潮玩手办延伸至文创产品，二手闲置物品、学习资料等多个领域也融合盲盒形成多元发展的局面。对于大学生而言，通过盲盒进行相互交换、购买，已逐渐成为校园里一种普遍的消费方式，同时也成为学生之间互动交流、增进情谊的重要社交载体。而微信小程序具有即用即走、无需安装、触达率高、开发成本低等特点，适合作为校园垂直应用的技术载体[1]。

#### 1.1.2 校园交易与配送现存痛点

盲盒由于其带来的惊喜感、不确定性和收集性受到很多年轻人的喜爱[2]，但是目前校园盲盒交易主要通过微信、QQ或者地摊进行。信息零散、价格不明、无保障等都是问题所在[3]。买家无法查看他人评价，卖家也很难宣传自己商品；此外，校园内最后一公里配送缺少有效的组织方式，一般情况下都是学生自愿帮忙拿取，反应迟缓而且效率不高，不能及时完成盲盒交易的需求[4]。大学内部二手闲置品数量较多，主要是书本、电子设备、日常用品等，由于缺乏有效的交换及捐赠途径而造成一定数量的闲置及浪费[5]。

学生有较强的文创创作意愿也缺少作品展示、品牌孵化以及交易变现的专属平台[6]。结合以上情况，使用微信小程序云开发架构设计并实现了一款集盲盒交易、二手交换、公益捐赠、文创IP、即时配送于一体的带有趣味互动功能的综合平台[7]。前端采用紫色霓虹灯风格，以营造科技感与神秘感。配送模块采用了宿舍楼分区抢单和1元统一定价策略，搭配摇一摇功能提供随机抽取盲盒的娱乐体验。

#### 1.1.3 平台建设的意义

综上所述，该小程序结合盲盒经济以及校园配送场景，既增加了移动设备上随机交互的设计实例，又给在校大学生提供了一个较为正规、便利、有趣的盲盒交易及配送平台，有利于推动校园内物品交流以及人际交往。

### 1.2 校园用户需求调研

#### 1.2.1 调研设计与样本情况

为了确定平台定位，进行基于武汉生物工程学院的网络调查及面对面访谈，在分层抽样的基础上共发出350份问卷，收回有效的占93.7%，而且每个年级的学生数量也大致相同。调查问题包括学生购买盲盒的习惯、是否愿意出售自己的闲置物品、对于配送的需求程度以及是否喜欢摇一摇等功能等。

#### 1.2.2 盲盒消费品类偏好分析

调研结果表明，学生群体对盲盒类型具有多元化的需求，其中文创手作类盲盒占35%最高，反映了学生对原创、个性化产品的强烈需求；闲置二手类盲盒占30%，说明二手物品盲盒化模式具备广泛的接受度[8]。IP联名类占20%，学习资料类占10%，其他类型占5%。具体数据如图1所示。

### 1.3 国内外研究现状

#### 1.3.1 国内研究现状

国内关于盲盒的研究多集中在消费心理学和营销策略方面[9]。微信小程序的开发与应用研究较为丰富，涵盖了二手交易、公益平台、学习工具等多个方向[10]。校园即时配送领域已有学者对配送调度算法进行了研究[11]。然而将盲盒交易、即时配送、二手交换、文创IP与摇一摇互动整合于一体的平台目前尚未见到相关研究报道。

#### 1.3.2 国外研究现状

国外"惊喜盒子"与电子商务平台融合发展较为完善，在此方面的研究主要集中在个性化推荐、用户购买行为分析及物流标准化建设等方面。但是专门面向校园场景的轻量级应用不多，这些产品物流成本高、模式较重，无法很好地服务于我国学校封闭式、低成本、碎片化配送需求。

#### 1.3.3 研究现状总结

综上所述，目前的研究工作已对小程序校园应用、盲盒购买、二手交易以及即时配送调度等方面进行了相关研究，但是仍未将它们结合在一起形成一个整体系统。因此本文从这一点出发进行研究。

### 1.4 研究内容与目标

#### 1.4.1 研究目标

本研究的目标是设计并实现一款满足交易、流转、配送、文创、公益一体化服务需求，且安全、稳定、易用、轻量化的校园盲盒即时配送微信小程序。

#### 1.4.2 主要研究内容

主要工作有：调查校园用户需求，确定功能及性能要求；设计云开发三层架构，在前端实现页面展示及操作；建立云函数、数据库以及云存储等构成的核心业务部分[12]；设计宿舍楼分区抢单调度算法并规定1元配送价格；进行系统测试及校内试运行，对平台进行检验是否可行并且稳定。

### 2.2 功能性需求

#### 2.2.1 功能需求

平台的核心业务功能需求主要包括以下几个方面：

**盲盒交易功能**：用户可以发布盲盒商品，设置价格、描述、图片等信息，管理员审核通过后上架销售；买家可以浏览盲盒商品列表，查看商品详情，进行下单购买和支付操作。

#### 2.2.2 非功能性需求

基于用户需求与技术可行性分析，系统需满足以下非功能需求：

**性能需求**：系统响应时间不超过2秒，支持同时在线用户数不少于100人，页面加载时间不超过1.5秒。

**可用性需求**：界面友好：采用统一的紫色霓虹灯设计风格；操作简单：操作流程简洁明了，任务完成率达到90%以上；响应式设计：适配不同尺寸的手机屏幕。系统可用性达到99.9%，支持7×24小时稳定运行，具备故障自动恢复能力。

**安全性需求**：安全需求包括：数据安全：用户敏感信息加密存储，数据传输采用HTTPS加密；交易安全：防止刷单、支付篡改等恶意行为；内容安全：对用户发布的商品内容进行审核；隐私保护：严格遵守用户隐私政策。

**易用性需求**：界面设计符合现代UI设计规范，操作流程简洁直观。

**可扩展性需求**：采用模块化设计，支持功能扩展与系统升级，可根据业务需求快速迭代。

### 2.3 可行性分析

**技术可行性**：微信小程序开发技术已经相当成熟，微信官方提供了完善的开发文档和开发者工具。微信云开发提供了Serverless架构，无需自建服务器，降低了开发难度。前端采用微信小程序原生开发，后端采用微信云开发，技术栈统一，开发效率高。

**经济可行性**：开发成本低：微信开发者工具、微信云开发均提供免费额度，无需购买服务器。运营成本低：微信云开发按量计费，初期用户量小，费用极低。经济效益预期：平台可收取交易佣金、广告收入、增值服务费等。

**操作可行性**：微信月活跃用户超过12亿，大学生群体几乎人手一个微信账号，无需额外注册。微信小程序的操作方式与微信原生应用一致，用户无需学习成本。通过前期调研，85%的受访学生表示愿意使用此类平台，用户接受度高。

**法律可行性**：平台仅提供信息发布和交易撮合服务，不直接参与交易，符合电商平台监管要求。用户协议明确平台责任边界，隐私政策符合《个人信息保护法》要求。

### 2.4 业务流程分析

系统的核心业务流程包括盲盒发布、购买、配送与自动捐赠四个环节：

**盲盒发布流程**：用户进入发布页面→填写盲盒标题、价格等基本信息→上传盲盒图片→选择发货与收货宿舍楼→编写或通过智能问答助手生成"时光小纸条"→提交发布请求→系统调用publishBox云函数→验证信息合法性→计算过期时间（7天）与捐赠时间（15天）→存储盲盒数据→返回发布结果。

**盲盒购买流程**：用户浏览盲盒商城→筛选或搜索目标盲盒→查看盲盒详情→点击购买按钮→确认订单信息（含配送费）→完成微信支付→系统调用createOrder云函数→生成订单记录→更新盲盒状态为已售出→返回购买结果。

**配送流程**：发布者确认发货→系统更新订单状态为待配送→骑手浏览待抢单列表→系统基于顺路得分算法推荐订单→骑手抢单→系统调用grabOrder云函数→更新订单状态为配送中→骑手取货→骑手送达→购买者确认收货→系统调用updateOrderStatus云函数→更新配送状态为已送达。

**自动捐赠流程**：系统每天凌晨2点触发定时任务→调用triggerAutoDonate云函数→查询15天无人处理的盲盒→批量更新盲盒状态为捐赠中→为每个盲盒创建捐赠记录→等待接收者领取→接收者确认领取→系统更新捐赠状态为已领取。

**代码清单2 自动捐赠定时任务代码**

```javascript
const cloud = require('wx-server-sdk')

cloud.init({ env: cloud.DYNAMIC_CURRENT_ENV })

const db = cloud.database()

const _ = db.command

exports.main = async (event, context) => {
  try {
    // 计算15天前的时间
    const fifteenDaysAgo = Date.now() - 15 * 24 * 60 * 60 * 1000
    
    // 查询需要自动捐赠的盲盒（发布超过15天仍在活跃状态）
    const boxesToDonate = await db.collection('boxes')
      .where({
        status: 'active',
        publish_time: _.lt(fifteenDaysAgo)
      })
      .get()
    
    // 批量处理自动捐赠
    for (const box of boxesToDonate.data) {
      // 更新盲盒状态为待捐赠
      await db.collection('boxes')
        .doc(box._id)
        .update({
          data: {
            status: 'donated_pending'
          }
        })
      
      // 添加捐赠记录
      await db.collection('donations')
        .add({
          data: {
            box_id: box._id,
            donor_id: box._openid,
            receiver_id: null,
            create_time: Date.now()
          }
        })
    }
    
    return { success: true, donatedCount: boxesToDonate.data.length }
  } catch (error) {
    console.error('自动捐赠失败', error)
    return { success: false, error: error.message }
  }
}
```

#### 4.3.4 即时配送算法实现

本系统提出一种基于几何距离的顺路得分算法，用于优化校园场景下的配送任务分配。该算法通过计算骑手的绕路距离比例，量化订单与骑手当前位置的顺路程度。

**算法数学模型**：

设骑手当前位置为R，盲盒发货宿舍楼位置为F，收货宿舍楼位置为T，则顺路得分的计算公式为：

```
顺路得分 = 1 - (|R→F| + |F→T| - |R→T|) / |R→T|
```

其中：
- |R→F|表示骑手到发货宿舍楼的距离
- |F→T|表示发货宿舍楼到收货宿舍楼的距离
- |R→T|表示骑手到收货宿舍楼的直线距离
- 分子部分表示绕路距离

当顺路得分≥0.8时，系统允许骑手抢单；当顺路得分<0.8时，系统判定为不顺路，拒绝抢单请求。

**代码清单3 顺路匹配算法代码**

```javascript
// 计算两点之间的距离（基于宿舍楼坐标）
function calculateDistance(point1, point2) {
  // 假设point1和point2为宿舍楼的坐标 {x, y}
  const dx = point1.x - point2.x;
  const dy = point1.y - point2.y;
  return Math.sqrt(dx * dx + dy * dy);
}

// 计算顺路得分
function calculate顺道Score(riderLocation, boxFromDorm, boxToDorm) {
  // 计算骑手到发货宿舍的距离
  const distanceToFromDorm = calculateDistance(riderLocation, boxFromDorm);
  // 计算发货宿舍到收货宿舍的距离
  const distanceFromTo = calculateDistance(boxFromDorm, boxToDorm);
  // 计算骑手到收货宿舍的直线距离
  const distanceToToDorm = calculateDistance(riderLocation, boxToDorm);
  // 顺路得分计算公式：1 - 绕路距离比例
  // 绕路距离 = 骑手到发货宿舍 + 发货宿舍到收货宿舍 - 骑手到收货宿舍
  const detourDistance = distanceToFromDorm + distanceFromTo - distanceToToDorm;
  // 直接距离 = 骑手到收货宿舍
  const directDistance = distanceToToDorm;
  // 计算绕路比例（确保分母不为0）
  const detourRatio = directDistance > 0 ? detourDistance / directDistance : 0;
  // 顺路得分，范围0-1，值越大表示越顺路
  const score = Math.max(0, Math.min(1, 1 - detourRatio));
  return score;
}

// 抢单逻辑
async function grabOrder(event) {
  const { orderId, riderOpenid, riderLocation } = event;
  try {
    // 查询订单信息
    const order = await db.collection('orders').doc(orderId).get();
    if (!order.data) {
      return { success: false, error: '订单不存在' };
    }
    // 检查订单状态
    if (order.data.delivery_status !== 'pending') {
      return { success: false, error: '订单已被抢' };
    }
    // 计算顺路得分
    const box = await db.collection('boxes').doc(order.data.box_id).get();
    const score = calculate顺道Score(riderLocation, box.data.from_dorm, box.data.to_dorm);
    // 只有顺路得分≥0.8的骑手才能抢单
    if (score < 0.8) {
      return { success: false, error: '不顺路，无法抢单' };
    }
    // 更新订单状态
    await db.collection('orders').doc(orderId).update({
      rider_openid: riderOpenid,
      delivery_status: 'delivering',
      updatedAt: new Date()
    });
    return { success: true, orderId, score };
  } catch (error) {
    console.error('抢单失败', error);
    return { success: false, error: error.message };
  }
}
```

#### 4.3.5 摇一摇互动模块实现

摇一摇抽盒是利用手机中的加速度传感器完成，调用wx.startAccelerometer获取加速度变化，interval设置为"ui"。根据X、Y、Z三个方向上的加速度绝对值总和判断是否摇动，而微信摇一摇建议范围是25至35之间，所以当大于30时视为一次有效的摇动。考虑到学校情况，在10名同学测试后认为设置为30可以去除走路或放在桌面上等情况造成的误操作，又能使用户正常使用时有较好的体验感。同时增加300ms的防抖动处理，避免短时间内出现多次触发。

**代码清单4 摇一摇加速度传感器监听代码**

```javascript
let lastShakeTime = 0;

wx.startAccelerometer({
  interval: 'ui',
  success: () => {
    wx.onAccelerometerChange((res) => {
      const now = Date.now();
      // 300ms节流，防止连续触发
      if (now - lastShakeTime < 300) return;
      // 计算三轴加速度总和
      const shake = Math.abs(res.x) + Math.abs(res.y) + Math.abs(res.z);
      if (shake > 30) {
        lastShakeTime = now;
        // 执行摇一摇抽盒逻辑
        this.doShakeDraw();
      }
    });
  }
});
```

针对当前校园盲盒交易缺少规范渠道、二手物品流转困难、末端配送效率偏低等问题，本文设计并实现了一款基于微信小程序云开发的校园盲盒即时配送与互动平台。

具体研究工作主要包括以下几个方面。首先通过在校园进行实地的需求调研，明确了盲盒交易、二手交换、即时配送、文创IP运营以及摇一摇互动这六大核心功能，完成了对用户需求的分析工作。

采用轻量化的云开发三层架构作为系统架构设计。前端是紫色霓虹灯风格的视觉界面搭配加速度传感器来完成摇一摇相关的交互操作，使用图片懒加载、本地缓存以及弱网环境优化等方法提升平台的性能；后端使用云函数与云数据库进行开发，实现了商品管理、订单处理、自动捐赠、配送结算以及摇一摇抽盒等一系列业务逻辑；针对校园配送特点，提出宿舍楼分区抢单与1元统一配送费的相关策略，还设计了适用于高峰期的顺路拼单优化算法，并且利用数据回测的方式有效验证该算法。

系统测试与校内试运行从可用性、稳定性和趣味性等多个维度验证了平台的实际运行效果。该平台将盲盒经济、共享经济、公益服务、趣味互动与校园生活相结合，为高校轻量化数字服务平台提供了一个可复用的实践参考。类似的校园服务平台设计在近年来也受到了广泛关注[15]。

### 6.2 不足与展望

#### 6.2.1 现存不足分析

平台的核心功能已经基本实现且通过了系统测试，但在研究进行的过程中仍发现有可以改进的地方。配送调度策略较为基础，高峰期难以对订单进行智能合并和路径优化，一定程度上限制了配送效率的进一步提高；摇一摇抽盒的概率参数是由管理员进行静态设定的，未引入用户的历史行为数据做个性化推荐，难以满足不同用户在互动偏好上的差异化需求；盲盒币的获取渠道只有每日签到和交易这两种方式，获取来源比较单一，缺少任务激励等多样化的机制，不利于提升用户的活跃度和留存率；平台售后纠纷处理流程与用户信用评价体系尚未完善，影响交易信任机制的建立工作；文创IP相关的功能还处在初步发展阶段，未达到从创作者入驻、内容孵化到价值转化的完整生态，文创元素所具有的吸引力未得到充分释放。

#### 6.2.2 未来优化方向

未来的优化方向主要包括以下几个方面：

1. **配送算法优化**：引入智能路径规划算法，实现订单自动合并和最优路径计算，提高配送效率。
2. **个性化推荐**：基于用户历史行为数据，实现盲盒内容和推荐的个性化，提升用户体验。
3. **多元化激励机制**：增加任务系统、成就系统等多种盲盒币获取渠道，提高用户活跃度。
4. **完善信任机制**：建立用户信用评价体系和售后纠纷处理流程，增强交易信任。
5. **文创生态建设**：拓展文创IP功能，构建从创作、孵化到变现的完整生态链。
6. **多平台适配**：考虑开发H5版本，支持更多设备和平台访问。
7. **数据安全加强**：进一步加强数据加密和隐私保护措施，确保用户数据安全。

通过持续的优化和迭代，平台有望成为校园内具有影响力的数字服务平台，为大学生提供更加便捷、有趣、安全的盲盒交易和配送服务。

```javascript
const cloud = require('wx-server-sdk')                     // 引入云开发SDK
cloud.init({ env: cloud.DYNAMIC_CURRENT_ENV })            // 初始化云开发环境

    const fifteenDaysAgo = Date.now() - 15 * 24 * 60 * 60 * 1000  // 计算15天前的时间戳
```

#### 4. 顺路匹配算法（deliveryService）

**文件路径：** `cloudfunctions/deliveryService/index.js`

```javascript
// 计算两点之间的距离（基于宿舍楼坐标）
function calculateDistance(point1, point2) {
  const dx = point1.x - point2.x
  const dy = point1.y - point2.y
  return Math.sqrt(dx * dx + dy * dy)
}

// 计算顺路得分
function calculate顺道Score(riderLocation, boxFromDorm, boxToDorm) {
  const distanceToFromDorm = calculateDistance(riderLocation, boxFromDorm)
  const distanceFromTo = calculateDistance(boxFromDorm, boxToDorm)
  const distanceToToDorm = calculateDistance(riderLocation, boxToDorm)
  const detourDistance = distanceToFromDorm + distanceFromTo - distanceToToDorm
  const directDistance = distanceToToDorm
  const detourRatio = directDistance > 0 ? detourDistance / directDistance : 0
  const score = Math.max(0, Math.min(1, 1 - detourRatio))
  return score
}

// 抢单逻辑
async function grabOrder(event) {
  const { orderId, riderOpenid, riderLocation } = event
  try {
    const order = await db.collection('orders').doc(orderId).get()
    if (!order.data) {
      return { success: false, error: '订单不存在' }
    }
    if (order.data.delivery_status !== 'pending') {
      return { success: false, error: '订单已被抢' }
    }
    const box = await db.collection('boxes').doc(order.data.box_id).get()
    const score = calculate顺道Score(riderLocation, box.data.from_dorm, box.data.to_dorm)
    if (score < 0.8) {
      return { success: false, error: '不顺路，无法抢单' }
    }
    await db.collection('orders').doc(orderId).update({
      rider_openid: riderOpenid,
      delivery_status: 'delivering',
      updatedAt: new Date()
    })
    return { success: true, orderId, score }
  } catch (error) {
    console.error('抢单失败', error)
    return { success: false, error: error.message }
  }
}
```
